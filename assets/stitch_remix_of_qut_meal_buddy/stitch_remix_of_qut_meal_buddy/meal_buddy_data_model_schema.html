# Meal Buddy: Data Model & Database Schema This document outlines the database structure required to support the Meal Buddy (QUT Edition) application, including user management, preferences, restaurant data, availability tracking, and matching logic. ## 1. ER Diagram Overview (Conceptual) The system is built around several core entities: - **Users**: Core student/staff data. - **Restaurants**: The dining locations. - **Preferences**: User-specific tastes and dietary needs. - **Availability**: User-defined time slots for meals. - **Selections**: Weekly restaurant choices (limit 3). - **Matches**: Successful connections between users. --- ## 2. Business Logic Constraints 1. **Selection Limit**: A database trigger or application logic must ensure `COUNT(weekly_selections)` where `user_id = X` and `week_number = Y` is <= 3. 2. **Matching Algorithm**: Matches are generated by joining `weekly_selections` and `user_availability` where `restaurant_id` and `time_slot` overlap. 3. **Availability Expiry**: Availability records should ideally be archived or cleared at the start of each new week to ensure fresh data.
